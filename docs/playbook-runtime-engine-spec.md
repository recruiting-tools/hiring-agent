# Playbook Runtime Engine — Specification

**Status:** ready for implementation  
**Depends on:** PR #29 (merged) — `management.playbook_sessions`, `management.playbook_steps`, `chatbot.vacancies`  
**Target service:** `services/hiring-agent`

---

## Context

The hiring-agent currently has three hardcoded playbooks in `src/playbooks/registry.js` with static execution logic. The playbook data model (definitions, steps, sessions) was defined and seeded in PR #29, but nothing reads or executes from it yet.

This task wires the DB-defined playbooks into the chat runtime.

---

## What to build

A **playbook runtime engine**: a stateful step executor that:

1. Reads `playbook_steps` from `management.playbook_definitions` / `management.playbook_steps`
2. Maintains session state in `management.playbook_sessions`
3. Executes steps by type (LLM call, DB fetch, show buttons, etc.)
4. Handles context variable substitution in `prompt_template`
5. Routes recruiter replies back to the correct waiting step

---

## Data flow

```
Recruiter types message / clicks chip
    ↓
routePlaybook(message, keywords from DB) → playbook_key
    ↓
runtime.dispatch({ playbook_key, vacancy_id, recruiter_input, ... })
    ↓
  load or create playbook_session
    ↓
  load current step from playbook_steps
    ↓
  execute step handler (by step_type)
    ↓
  update session context + current_step_order
    ↓
return reply to UI (buttons / text / display / loading)
```

---

## Session lifecycle

### Start
- Recruiter selects a vacancy (UI sets `vacancy_id` in request body alongside `job_id`)
- Recruiter triggers a playbook (keyword or chip click → `playbook_key`)
- Runtime checks `management.playbook_sessions` for an active session:
  ```sql
  SELECT * FROM management.playbook_sessions
  WHERE tenant_id = $1 AND recruiter_id = $2
    AND vacancy_id = $3 AND playbook_key = $4
    AND status = 'active'
  LIMIT 1
  ```
- If not found → create new session, `current_step_order = 0`
- If found → resume at `current_step_order`

### Advance
- After each step completes: `UPDATE playbook_sessions SET current_step_order = next_step_order, context = $updated, updated_at = now()`
- If `next_step_order IS NULL` → set `status = 'completed'`, `completed_at = now()`

### Abort
- Recruiter types `/cancel` or starts a different playbook → set `status = 'aborted'`

## Step type handlers

### `auto_fetch` (step_order = 0, always silent)
Load vacancy from tenant DB, inject into session context. Never shown to recruiter.

```js
const vacancy = await tenantSql`
  SELECT * FROM chatbot.vacancies WHERE vacancy_id = ${context.vacancy_id}
`[0];
context.vacancy = vacancy;
context.raw_vacancy_text = vacancy.raw_text;
// advance immediately to step 1, no reply to UI
```

### `buttons`
Show `user_message` + option buttons from `step.options` (semicolon-separated).

Reply shape:
```json
{
  "kind": "buttons",
  "message": "step.user_message",
  "options": ["Вариант 1", "Вариант 2"],
  "step_key": "create_vacancy.3"
}
```

On next recruiter message: match against options, store in `context[context_key]`, advance to `next_step_order`.

### `user_input`
Show `user_message`. Wait for free text.

Reply shape:
```json
{ "kind": "user_input", "message": "step.user_message" }
```

On next message: store `recruiter_input` in `context[context_key]`, advance.

### `llm_extract`
Render `prompt_template` → call LLM → parse JSON → store in `context[context_key]`.  
If `db_save_column` is set → UPDATE `chatbot.vacancies` column.

```js
const prompt = interpolate(step.prompt_template, context);
const raw = await llm.generate(prompt);
const parsed = JSON.parse(raw);
context[step.context_key] = parsed;
if (step.db_save_column) {
  await tenantSql`
    UPDATE chatbot.vacancies
    SET ${sql(step.db_save_column)} = ${JSON.stringify(parsed)}
    WHERE vacancy_id = ${context.vacancy_id}
  `;
}
// advance silently (no reply to UI — next step renders confirmation)
```

### `llm_generate`
Render `prompt_template` → call LLM → return text/HTML to UI.

Reply shape:
```json
{ "kind": "llm_output", "content": "...", "content_type": "html|text" }
```

Store result in `context[context_key]`. Advance to `next_step_order`.

### `display`
Show pre-built content from `context[context_key]` (already generated by prior llm step).  
Optional continue/refine buttons from `step.options`.

Reply shape:
```json
{
  "kind": "display",
  "content": "context[context_key]",
  "content_type": "html",
  "options": ["Подтвердить", "Изменить"]
}
```

### `decision`
Evaluate a condition on context, route to different `next_step_order`.

The routing rules are encoded in `step.notes` as JSON (for now). Example:
```json
{
  "rules": [
    { "condition": "context.must_haves.length < 2", "next": 2, "message": "Нашли мало обязательных требований..." },
    { "condition": "context.must_haves.length >= 5", "next": 2, "message": "Нашли много обязательных требований..." },
    { "default": true, "next": 4 }
  ]
}
```

No UI reply if silent routing. Shows message if rule has one.

### `subroutine`
Push `{playbook_key, return_step_order}` to `call_stack`. Start sub-playbook at step 0.

On sub-playbook `completed`: pop from `call_stack`, resume parent at `return_step_order`.

### `data_fetch`
Execute a parameterized SQL query on `tenantSql`. Store result in `context[context_key]`. Used by `candidate_funnel` playbook.

SQL is in `step.notes` with `{{context.vacancy.vacancy_id}}` substitution.

## Context variable interpolation

Function: `interpolate(template, context) → string`

Syntax: `{{path}}` with optional filter `{{path | filter}}`

Paths use dot notation: `context.vacancy.must_haves`

Filters:
| Filter | Output |
|--------|--------|
| `bullet_list` | `• item1\n• item2` |
| `formatted` | JSON pretty-print of object |
| `json` | `JSON.stringify` |
| `names_only` | Array of objects → join `.name` fields |
| `in_scope_only` | Filter array where `in_our_scope === true`, then `names_only` |

## `runtime.dispatch()` interface

```js
// services/hiring-agent/src/playbooks/runtime.js

export async function dispatch({
  managementSql,    // postgres connection to management DB
  tenantSql,        // postgres connection to tenant (chatbot) DB
  tenantId,         // string
  recruiterId,      // string
  vacancyId,        // string — comes from UI request body
  playbookKey,      // string — resolved by routePlaybook()
  recruiterInput,   // string | null — text typed by recruiter
  llmAdapter,       // { generate(prompt) → string }
}) → { reply: ReplyShape, sessionId: string }
```

---

## Reply shapes (union)

```ts
type Reply =
  | { kind: "buttons";    message: string; options: string[]; step_key: string }
  | { kind: "user_input"; message: string }
  | { kind: "llm_output"; content: string; content_type: "html" | "text" }
  | { kind: "display";    content: string; content_type: "html" | "text"; options?: string[] }
  | { kind: "completed";  message: string }
  | { kind: "fallback_text"; text: string }
  // existing:
  | { kind: "render_funnel"; ... }
```

## Routing change

Current: `routePlaybook(message)` uses regex patterns hardcoded in `router.js`.

New: load keywords from DB at startup (or first request), match against `playbook_definitions.keywords TEXT[]`.

```js
// On startup or first use:
const defs = await managementSql`
  SELECT playbook_key, keywords FROM management.playbook_definitions
  WHERE status = 'available'
`;
// keywords is TEXT[] — check if any keyword is substring of message
```

Chip clicks (UI sends `{ action: "start_playbook", playbook_key: "create_vacancy" }`) bypass text routing entirely.

## LLM adapter

Use existing `GeminiAdapter` from `candidate-chatbot`. Either import directly or extract to shared package.

For `llm_extract`: prompt must end with instruction `Return valid JSON only, no markdown.` Add this automatically in the handler if not already present in `prompt_template`.

LLM errors: retry once, then set session `status = 'error'`, return user-friendly message.
